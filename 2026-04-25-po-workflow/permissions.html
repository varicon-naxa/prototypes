Value increase When a PO's total is increased, the change goes through the same value-band rules at the new total. Only the highest tier triggered by the new total re-approves — earlier tiers don't redo, since they already vouched for the deal at the lower value.

While the change is pending, the PO's approved value stays at the prior amount and bills can keep being raised against the approved-value headroom. The approved value updates only when the change is approved. Reject clears the pending change with no effect on the PO.
Close When total billed equals the approved PO value, the system surfaces a soft prompt asking whether to close the PO. Manual short-close for under-deliveries is available at any time, performed by anyone in the PO's approval chain. Cancel is a separate action for un-actioned POs.

Removed from the previous workflow

  • Can Create / Submit tier — anyone with PO module access can draft. The approval gate is the only value-based control. (No reason to gate creation when approval already gates value.)
  • Can Close tier — close is an action available to anyone in the PO's approval chain, not a separate permission to configure.
  • Increment / Decrement amount fields per approver — replaced by the pending-change model: edit the PO to a higher value and the change goes through approval as a discrete pending layer.
Open decisions for review:
  • Sequential vs parallel within a tier — recommended sequential (junior → senior → exec)
  • Manual close permission scope — recommended: anyone in the PO's approval chain
  • Cancel as a distinct action with lighter approval — recommended yes (creator + one approver in chain)

Edit value band
